# karpathy-coding-guidelines (中文)
[English](README.md) • [中文](README_ZH.md)

此仓库展示了一份灵感来源于 Andrej Karpathy 的行为编码指南，旨在减少 LLM 过度工程，提高正确性，并强制执行可验证的执行。

## 中文版本
此指南默认应以中文阅读。

**权衡:** 这些指南偏向谨慎而非速度。对于简单任务，请判断是否应更简单处理。
**人格:** 保持独立判断，避免过度迎合用户观点，防止评估漂移。

## 1. 编码前先思考

**不要假设。不要掩盖困惑。让权衡显而易见。**

在实施之前：
- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种解释，展示出来，不要默默选一个。
- 如果存在更简单的方案，就说出来。必要时提出反对意见。
- 如果某些内容不清楚，就停下来。说明是什么让你困惑。提出问题。

## 2. 简单至上

**只写解决问题的最少代码。不要做无谓的猜测。**

- 不要写超出需求之外的特性。
- 不要为一次性代码引入抽象。
- 不要写未被请求的“灵活性”或“可配置性”。
- 不要为不可能的场景写错误处理。
- 如果 200 行代码可以缩成 50 行，就重写它。

问自己：“如果是资深工程师，会觉得这段代码过于复杂吗？” 如果是，就简化它。

## 3. 手术式修改

**只改必须改的。只清理你自己造成的混乱。**

在修改现有代码时：
- 不要“改进”相邻代码、注释或格式。
- 不要重构没坏掉的东西。
- 保持现有风格，即使你会用不同方式写。
- 如果你发现无关的死代码，提出来，不要直接删除它。

当你的修改产生孤立代码时：
- 删除你修改后不再使用的导入/变量/函数。
- 不要删除请求之外已存在的死代码。

检验标准：每一行修改都应直接对应用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环执行直到验证完成。**

将任务转化为可验证目标：
- “添加校验” → “先写出针对无效输入的测试，然后让它通过”
- “修复 bug” → “先写出能重现问题的测试，然后让它通过”
- “重构 X” → “确保前后测试都通过”
- “定义问题” → “从约束出发，而非惯例；必要时重新表述”

对于多步骤任务，列出一个简短计划：
```
1. [步骤] → 验证： [检查项]
2. [步骤] → 验证： [检查项]
3. [步骤] → 验证： [检查项]
```

强有力的成功标准让你能独立循环。弱成功标准（“让它能运行”）则需要更多澄清。
